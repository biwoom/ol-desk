# 1. 한 줄 정의

**번역워크플로우 운영매뉴얼은 원문 수집부터 1차·2차 번역, OL-DESK 인간감수, 3차 반영, 4차 최종원고, revision cycle까지의 실제 작업 순서를 정의한다.**

---

# 2. 전체 워크플로우

OL-DESK-CREW의 번역 워크플로우는 네 종류로 나눈다.

```text
1. 1-2차 번역 워크플로우
2. 2차 휴먼 워크플로우
3. 3차 번역 워크플로우
4. 3차 번역본 및 4차 원고 수정 워크플로우
```

각 워크플로우의 주도권은 다르다.

```text
1-2차 번역 워크플로우
= A라인 주도

2차 휴먼 워크플로우
= OL-DESK + 인간 감수자 주도

3차 번역 워크플로우
= B라인 주도

4차 원고 수정 워크플로우
= OL-DESK + 인간 감수자 주도
```

# 4. 1-2차 번역 워크플로우

## 4.1 주도

```text
A00-2차 생산 관리 매니저-선행
```

## 4.2 순서

```text
source_ready
→ source_normalized
→ reference_ready 또는 reference 없음
→ term_candidates_done
→ draft1_done
→ source_review_done
→ annotation_candidates_done
→ draft2_done
→ human_review_ready
```

## 4.3 단계별 완료 조건

### source_ready

```text
필수:
- assets/source/raw/{doc_id}.md 존재
- data/documents.json에 문서 등록

담당:
- A00 또는 A01
```

### source_normalized

```text
필수:
- assets/source/normalized/{doc_id}.md 존재
- data/segments/{doc_id}.json 존재
- A01 agent-log 존재
```

### reference_ready

```text
필수:
- 참고번역이 있는 경우 assets/references/{doc_id}.md 존재
- A02 agent-log 존재
- 본문 삽입 각주가 있는 경우 Markdown 각주 형식으로 정리되어 있음

참고번역이 없으면 이 상태를 건너뛸 수 있다.
```

### term_candidates_done

```text
필수:
- data/terms/terms.json에 후보 추가 또는 후보 없음 명시
- A03 agent-log 존재

주의:
- 후보는 approved 상태가 될 수 없다. 인간 확정 전에는 candidate 또는 hold 상태로 둔다.
- 기존 용어집이 있는 경우 기존 approved 용어를 먼저 확인한다.
- 기존 용어집에 없는 신규 용어만 후보로 추가하고, 대안 번역어와 충돌 가능성을 기록한다.
```

### draft1_done

```text
필수:
- outputs/manuscripts/draft1/{doc_id}.md 존재
- A04 agent-log 존재
```

### source_review_done

```text
필수:
- A05 agent-log 존재
- 원문 누락, 의미 추가, 오역 후보 점검 결과 기록
```

### annotation_candidates_done

```text
필수:
- data/annotations/annotations.json에 후보 추가 또는 후보 없음 명시
- A06 agent-log 존재

주의:
- A06은 각주 후보만 생성한다.
- 본문 해석 후보나 교리 해설문을 생성하지 않는다.
```

### draft2_done

```text
필수:
- outputs/manuscripts/draft2/{doc_id}.md 존재
- A07 agent-log 존재
- data/status/{doc_id}.json 업데이트
```

### human_review_ready

```text
필수:
- B01 agent-log 존재
- data/revisions/{doc_id}.json에 revision 초안 생성
- OL-DESK에서 검수대기 상태 표시 가능
```

# 5. 2차 휴먼 워크플로우

## 5.1 주도

```text
OL-DESK + 인간 감수자
```

## 5.2 순서

```text
human_review_ready
→ human_editing
→ draft3_handoff_ready
```

## 5.3 인간 감수자가 하는 일

```text
- 원문 확인
- 1차 번역 보기
- 2차 번역 보기
- 병렬보기로 의미 비교
- 3차 가번역 직접 수정
- 용어 후보 채택·보류·폐기
- 각주 후보 채택·보류·폐기
- 태그 후보 채택·보류·폐기
- 교정·검수 메모 작성
- 신규 용어 생성
- 신규 각주 생성
- 신규 태그 생성
- draft3 handoff JSON 생성
```

## 5.4 3차 가번역 원칙

```text
초기값:
- 2차 번역본

수정 가능:
- 세그먼트별 번역문
- 용어
- 각주
- 태그
- revision note
- 감수자 메모

저장:
- 원고는 revision JSON의 modified_segments 또는 별도 draft3 preliminary 저장 구조에 기록
- 확정 용어·각주·태그는 각 JSON에 approved 상태로 반영
- 감수자 메모는 data/notes/{doc_id}.json에 저장
```

감수자 메모는 본문 해석 메모가 아니라 교정·검수 메타데이터이다. 메모는 자동으로 본문이나 handoff에 반영되지 않으며, 인간 감수자가 선택한 메모만 handoff에 연결한다.

## 5.5 draft3_handoff_ready 완료 조건

```text
필수:
- handoff/draft3/{doc_id}.json 존재
- data/revisions/{doc_id}.json 업데이트
- 인간 확정 항목과 보류 항목이 구분되어 있음
- AI-Agent 참고용으로 선택된 메모만 handoff에 포함되어 있음
```

handoff에 없는 후보는 B라인이 확정 반영하지 않는다.

# 6. 3차 번역 워크플로우

## 6.1 주도

```text
B00-3차 확정 관리 매니저-결정
```

## 6.2 순서

```text
draft3_handoff_ready
→ draft3_requested
→ draft3_generated
→ final_scan_done
→ final_review
```

## 6.3 단계별 완료 조건

### draft3_requested

```text
필수:
- B00이 handoff/draft3/{doc_id}.json 확인
- B02 작업 배정
- data/status/{doc_id}.json 업데이트
```

### draft3_generated

```text
필수:
- outputs/manuscripts/draft3/{doc_id}.md 존재
- B02 agent-log 존재
- 인간 확정 항목만 반영됨
```

### final_scan_done

```text
필수:
- B03 agent-log 존재
- 원문 누락 여부 확인
- handoff 반영 누락 여부 확인
```

### final_review

```text
필수:
- OL-DESK에서 4차 원고 탭 편집 가능
- outputs/manuscripts/draft4/{doc_id}.md 초안 또는 인계본 존재
- B04 agent-log 존재
```

# 7. 4차 최종원고 워크플로우

## 7.1 주도

```text
OL-DESK + 인간 감수자
```

## 7.2 순서

```text
final_review
→ final_done
```

## 7.3 인간 감수자가 하는 일

```text
- 3차 번역본 확인
- 4차 최종원고 직접 수정
- 최종 문체 조정
- 최종 용어 확인
- 최종 각주 확인
- 최종 태그 확인
- 최종 교정·검수 메모 확인
- 최종 승인
- 개별·전체 Markdown 내보내기
```

## 7.4 final_done 완료 조건

```text
필수:
- outputs/manuscripts/draft4/{doc_id}.md 존재
- data/status/{doc_id}.json의 current_status가 final_done
- 인간 최종 승인 기록 존재
- 필요한 경우 outputs/exports/individual/ 또는 outputs/exports/all/에 Markdown 내보내기 산출물 생성
```

final_done은 출판 완료가 아니다. OL-DESK-CREW 안에서 최종 원고가 승인된 상태를 뜻한다.

# 8. Revision Cycle

완료본 수정은 과거 상태로 단순 회귀하지 않는다. 새 revision cycle을 연다.

## 8.1 순서

```text
final_done
→ revision_needed
→ draft3_preliminary_reopened
→ draft3_handoff_ready
→ draft3_requested
→ draft3_generated
→ final_scan_done
→ final_review
→ final_done
```

## 8.2 Revision 발생 조건

```text
- final_done 이후 본문 수정 필요
- 확정 용어 변경
- 각주 수정
- 태그 수정
- 원문 누락 또는 오역 발견
- 문체 정책 변경
```

## 8.3 Revision 기록

필수 기록:

```text
- data/revisions/{doc_id}.json에 새 revision 추가
- handoff/revision/{doc_id}-{revision_id}.json 생성
- 변경 전후 Git diff 확인 가능
- data/status/{doc_id}.json 업데이트
```

# 9. 보류와 오류

## 9.1 blocked

`blocked`는 외부 판단이나 원문 확인 없이는 진행할 수 없는 상태이다.

예:

```text
- 원문 출처 불명확
- 참고번역과 원문 혼입
- 문단 구조 불명확
- 인간 판단 필요
```

## 9.2 error

`error`는 단계 무결성이 깨진 상태이다.

예:

```text
- 필수 파일 없음
- JSON 구조 오류
- handoff와 3차 번역본 불일치
- 에이전트 로그 누락
- 잘못된 단계 건너뛰기
```

## 9.3 처리 원칙

```text
blocked는 이유와 필요한 판단을 기록한다.
error는 재현 가능한 오류 내용과 수정 담당을 기록한다.
둘 다 임의로 다음 단계로 넘기지 않는다.
```

# 10. 상태 전환 금지 규칙

```text
source_ready 없이 draft1_done으로 갈 수 없다.
draft1_done 없이 source_review_done으로 갈 수 없다.
source_review_done 없이 draft2_done으로 갈 수 없다.
draft2_done 없이 human_review_ready로 갈 수 없다.
human_editing 없이 draft3_handoff_ready로 갈 수 없다.
handoff 없이 draft3_generated로 갈 수 없다.
draft3_generated 없이 final_review로 갈 수 없다.
인간 승인 없이 final_done으로 갈 수 없다.
```

# 11. OL-DESK 표시 원칙

OL-DESK는 상태를 숨기지 않는다.

```text
- 현재 상태
- 다음 작업
- 담당 라인
- 마지막 agent-log
- 인간 검수 필요 여부
- 보류·오류 사유
- revision cycle 여부
```

진행 페이지에서는 간략히 표시하고, 번역 페이지에서는 상세 표시한다.

---

# 12. 감수자 메모 운영 원칙

감수자 메모는 인간 감수자가 스스로를 위해 남기는 교정·검수 기록이다.

```text
입력:
- 감수자는 댓글처럼 메모 본문만 입력한다.

자동 연결:
- project_id
- doc_id
- 현재 페이지
- 현재 탭
- 현재 단계
- segment_id 또는 target_id
- 작성 시각

저장:
- data/notes/{doc_id}.json

금지:
- 메모를 본문에 자동 삽입하지 않는다.
- 메모를 용어·각주·태그 승인으로 자동 처리하지 않는다.
- 메모를 handoff에 자동 포함하지 않는다.
```

메모는 `open`, `resolved`, `archived` 상태를 가진다.

---

# 13. Markdown 내보내기 운영 원칙

Markdown 내보내기는 승인된 번역본을 사람이 읽기 쉬운 산출물로 생성하는 단계이다.

```text
개별 내보내기:
- outputs/exports/individual/{doc_id}-{stage}.md

전체 내보내기:
- outputs/exports/all/{project_id}-{stage}-all.md
```

내보내기는 원고의 파생 산출물이며, 수정정보·상태·후보·로그·메모 JSON을 대체하지 않는다.

# 14. 최종 원칙

```text
A라인은 2차 번역까지 생산한다.
OL-DESK는 인간 감수와 3차 가번역 수정을 담당한다.
B라인은 인간 handoff만 반영해 3차 번역을 생성한다.
4차 최종원고는 인간이 수정하고 승인한다.
감수자 메모는 JSON으로 저장하고, 댓글형 UX로 간단하게 작성한다.
개별·전체 번역본 내보내기 산출물은 Markdown으로 생성한다.
완료본 수정은 새 revision cycle로 처리한다.
모든 상태 전환은 필수 파일과 agent-log로 증명되어야 한다.
```
